# 材料哈希预留 fencing 修复报告

## 状态

已修复剩余的材料哈希预留一致性与并发持久化问题。

- 启动协调现在覆盖全部预留状态：已有材料按 `createdAt`、`id` 选定规范所有者并标记为 `committed`；没有对应材料的 `committed` 孤儿会被删除；没有对应材料的过期 `pending` 会被删除；未过期 `pending` 保留。
- 历史重复材料不会被删除，仅修正规范预留指向。
- 材料、分块、事实以及预留从 `pending` 到 `committed` 的转换现在位于同一个 SQLite 事务中。
- 事务通过 `contentHash + materialId + pending` 条件更新并使用 `RETURNING` 校验恰好一行；租约被接管的旧请求无法写入任何数据库记录，只清理自己的上传文件并返回错误。
- 任一材料数据插入失败会回滚预留提交及全部材料数据，避免数据库记录指向已清理的文件。
- 获取预留时，若 UPSERT 冲突后的 SELECT 恰逢所有者释放导致记录消失，会重新尝试获取，最多三次。
- 保留显式 `in_progress` 409、仅所有者释放、清理 `AggregateError`、启动协调、删除后规范所有者重分配及 UI 成功刷新行为。

## TDD 证据

先增加并运行失败测试，观察到三个预期红灯：`committed` 孤儿未删除，以及事务化 fencing 持久化函数尚不存在的两个失败。实现最小事务修复后，预留测试 7/7 通过。

随后先移除测试依赖中的独立提交回调，观察到六个测试因生产代码仍调用 `commitContentHash` 而失败；再删除生产代码中的独立提交路径并把路由接入事务化持久化，相关测试恢复为绿色。

真实 SQLite 回归覆盖：

- 旧所有者模型处理延迟到租约被新所有者接管后，旧所有者持久化失败，新所有者成功，最终仅有新材料及新模型事实。
- 分块插入发生主键冲突时，材料、分块和预留提交全部回滚，预留仍为原所有者的 `pending`。
- 全状态启动协调正确删除 `committed` 孤儿与过期 `pending`，保留未过期 `pending` 和全部历史材料。

## 关注点

- fencing 发生在数据库事务边界；文件写入仍先于数据库事务，因此所有权丢失或事务失败时依赖现有清理流程删除当前请求自己的上传目录。该清理保持所有者隔离，并在清理不完整时继续通过 `AggregateError` 暴露。
- E2E 启动时仍有 Next.js 多 lockfile 根目录推断警告以及 `NO_COLOR`/`FORCE_COLOR` 警告，均为既存环境警告，不影响 6 个 E2E 用例通过。
- 未调用 DashScope。
