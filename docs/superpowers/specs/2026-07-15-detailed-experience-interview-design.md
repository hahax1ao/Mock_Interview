# 详细经历画像与项目面试设计

## 概要

在现有普通画像事实之外，引入独立的“详细经历卡片”，完整保存科研、项目和竞赛经历的名称、背景、职责、方法、成果与逐字段证据。模拟面试的科研项目模块必须优先围绕已确认经历卡片提问，避免详细项目被普通事实数量限制、材料片段截断或 PDF 阅读顺序干扰。

首版继续使用本地文件、SQLite、Drizzle ORM 和阿里云百炼 DashScope。原始文件不发送给百炼，只发送本地提取出的相关文本。默认测试全部使用 mock；真实百炼验收必须由用户单独授权。

## 问题与根因

`jianli.pdf` 的本地 PDF 解析已经完整提取 Super-LoRa 内容，包括研究目标、STM32/SX1276 驱动、Pluto SDR 链路、CFO 补偿、De-chirp 解调、复合信道和量化指标。遗漏发生在智能画像与面试上下文阶段：

- 当前智能提取把经历压缩为最长 500 字的一条普通画像事实，详细项目容易被模型省略。
- 普通事实不能表达“同一经历下的背景、职责、方法与成果”之间的归属关系。
- 面试上下文只取前 20 条已确认事实，详细经历可能排在限制之外。
- 材料片段最多截取 800 字，长项目的后半部分和量化结果可能无法进入面试上下文。
- PDF 文本阅读顺序可能把栏目标题移动到页面其他位置，不能依靠相邻标题判断经历类型。

## 数据模型

新增 `profile_experiences` 表，每一行表示一张经历卡片：

| 字段 | 含义 |
| --- | --- |
| `id` | 经历卡片标识 |
| `materialId` | 来源材料，删除材料时级联删除 |
| `type` | `research`、`project` 或 `competition` |
| `title` | 经历名称 |
| `background` | 背景与目标，可为空 |
| `responsibilities` | 候选人的个人职责，可为空 |
| `methods` | 技术方法、工具和实现过程，可为空 |
| `results` | 实验结果和量化指标，可为空 |
| `awardRole` | 奖项、名次、队内角色，可为空 |
| `source` | 原材料名称 |
| `page` | 真实页码 |
| `evidence` | `{title, background?, responsibilities?, methods?, results?, awardRole?}` 形式的逐字证据 JSON |
| `confidence` | 综合置信度 |
| `status` | `draft` 或 `confirmed` |
| `createdAt`、`updatedAt` | 创建与更新时间 |

经历去重键为“材料 ID、类型、标准化名称”。重复智能提取更新未确认草稿，不新增重复卡片；已确认卡片永不被自动覆盖。

普通 `profile_facts` 继续保存排名、成绩、课程、英语、技能、荣誉等扁平事实。科研、项目和带详细项目描述的竞赛内容写入经历卡片。竞赛奖项既可保留为普通荣誉事实，也可作为详细竞赛卡片的 `awardRole`，两者用途不同，不视为重复。

## 智能提取

百炼结构化输出从扁平 `facts` 扩展为 `experiences`。每张经历包含类型、名称、五类可选详情、页码、置信度和逐字段证据。规则如下：

- 按语义识别经历，不依赖 PDF 中栏目标题的位置。
- 经历名称必填；背景、职责、方法、成果和奖项角色至少有一项非空。
- 每个非空字段必须包含同页逐字证据，证据经 Unicode 和空白归一化后必须存在于该页原文。
- 模型不得把不同经历的内容合并，也不得把课程、成绩、英语成绩或联系方式包装成经历。
- 返回部分有效字段时保存为不完整草稿，并在界面提示缺失字段；名称无效、所有详情为空、页码无效或证据不可定位的卡片拒绝落库。
- Super-LoRa 应生成一张科研经历卡片，完整关联研究目标、个人实现、算法方法和量化性能，而不是拆成互不关联的普通事实。
- 识别规则不得硬编码 Super-LoRa、嵌入式比赛或电赛等具体名称。材料中任何具有独立名称，并包含背景、个人职责、技术方法、实现过程、量化结果或奖项角色等详细描述的科研、项目或竞赛经历，都必须生成独立卡片。
- 全国大学生嵌入式芯片与系统设计竞赛 FPGA 项目应生成一张竞赛经历卡片，保存智能会议相机的背景、队长和 FPGA/上位机职责、硬件与软件实现以及丢包率指标。
- 全国大学生电子设计竞赛 G 题项目应生成另一张竞赛经历卡片，保存电路模型探究装置、队长和 FPGA/硬件职责、RLC 网络识别方法与实现过程。

现有个人材料上传流程仍先完成本地解析和普通事实提取，再调用百炼生成经历卡片。智能调用失败时材料继续保留本地索引与普通事实，不删除旧卡片。

## API 与写入规则

新增或扩展以下接口：

- `GET /api/materials`：返回材料、普通事实和经历卡片。
- `POST /api/materials/:id/retry`：即使材料已经完成普通智能解析，也允许重新提取详细经历。
- `POST /api/experiences/:id/confirm`：使用 Zod 校验整张编辑后的卡片，并原子更新所有字段及 `confirmed` 状态。
- `PATCH /api/experiences/:id`：保存草稿字段；对已确认卡片调用时先把状态恢复为 `draft`，再显式保存用户修改。

写入智能提取结果时，在一个 SQLite 事务内完成草稿去重、更新和新增。事务不得覆盖 `confirmed` 卡片。材料删除沿用现有安全删除流程，并级联删除经历卡片；删除失败仍执行隔离与回滚。

## 画像界面

普通画像事实区域保持现状，并在其下增加“详细经历”区域：

- 每张经历以可展开卡片显示，标题栏展示类型、名称、置信度和来源页码。
- 展开后显示并允许编辑背景目标、个人职责、技术方法、量化成果和奖项角色。
- 用户修改字段后点击一次“整张确认”；确认后默认只读，并提供“重新编辑”。
- 不完整草稿明确标出缺失字段，但允许用户补全后确认。
- 已完成解析的旧材料也显示“重新提取详细经历”，无需删除重传。
- 重新提取成功后刷新卡片；失败时保留现有内容并显示可重试错误。

## 面试出题

创建面试时仅使用用户所选材料。普通事实仍只有已确认内容才能作为确定事实；详细经历同样只有 `confirmed` 卡片才能成为强制项目题依据。

科研项目导师模块开始前，应用根据以下顺序选择一张核心经历：

1. 与用户配置的面试重点相关度。
2. 经历字段完整度。
3. 是否包含量化成果。
4. 稳定的创建时间和 ID 顺序，保证结果可测试。

只要存在已确认经历卡片，10、20、30 分钟场次的科研项目模块都必须先围绕一张核心经历提问。应用在角色交接控制指令中明确给出核心经历和首问要求，不能只依赖基础提示词自由选择。

第一问必须点名该经历，后续按“动机与职责、技术方法、实验结果、局限与改进”自适应追问，并遵守同一主题最多连续追问三层。之后才允许切换到其他经历或通用科研问题。

经历卡片以紧凑结构化文本放在材料上下文最前方，不受普通事实前 20 条和原文片段 800 字截断影响。原文片段仅作为补充证据。实时语音、文字降级和复盘评审共用同一经历上下文。没有已确认卡片时，系统退回当前普通事实与材料片段逻辑。

## 错误处理与隐私

- 百炼失败、超时或结构校验失败不影响本地材料索引、普通画像事实和已有经历卡片。
- 重试失败显示明确错误并允许再次重试。
- 不完整但有效的卡片保存为草稿，不把空字段补写成模型推测。
- 用户确认内容优先于后续模型输出。
- 原始 PDF、DOCX 和图片只保存在本机；百炼仅接收本地提取文本。
- 日志不记录 API Key，也不输出完整个人材料文本。

## 测试与验收

- 智能提取单元测试使用包含完整 Super-LoRa、嵌入式 FPGA 比赛和电子设计竞赛 G 题描述的固定文本，验证三张独立卡片均被保留，并验证识别逻辑不依赖这三个具体名称。
- 验证每个非空字段的页码与逐字证据；证据不存在、跨页错配和空详情卡片必须被拒绝。
- 数据库测试覆盖表迁移、草稿新增、重复解析更新、确认卡片保护、整卡确认和材料级联删除。
- API 测试覆盖已完成材料重新提取、百炼失败保留旧数据和非法确认请求拒绝落库。
- 界面测试覆盖卡片展开、编辑、缺失提示、整卡确认、重新编辑和重新提取。
- 面试上下文测试验证已确认 Super-LoRa 卡片优先出现，未确认卡片不被当作确定事实，详细方法与量化指标不被截断。
- 角色交接测试验证科研模块的首问控制指令包含被选中的核心经历。
- 默认 Vitest 和 Playwright 全部使用 mock，不调用百炼。真实 `jianli.pdf` 验收仅在用户明确授权发送其本地解析文本后手动运行。

## 首版范围

本次只实现详细经历的提取、管理和面试优先使用。暂不实现跨材料自动合并同一经历、经历版本历史、用户自定义经历类型、公共项目题库或联网核验论文与竞赛信息。
